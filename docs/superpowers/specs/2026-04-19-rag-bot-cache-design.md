# Дизайн: RAG в боте без блокировки event loop + корректный ключ кэша

**Дата:** 2026-04-19  
**Статус:** утверждено пользователем  
**Волна:** D — короткий сквозной релиз (минимум A + B)

## 1. Цель и границы

### Цель

- Не блокировать asyncio event loop процесса бота на время выполнения RAG (эмбеддинги, Chroma, реранк, LLM).
- Исключить попадание в кэш ответов, сгенерированных другим LLM backend, при смене `backend` в `execute_rag_query`.
- Сохранить текущую семантику `RAGPipeline.query` и цепочку ранжирования без изменений.

### Вне объёма (не в этой волне)

- Параллельные запросы к ChromaDB, оптимизация `count_chunks_for_channel`, ingest.
- Таймауты на весь RAG-вызов.
- Замена `print` на `logger` в `chromadb_store` и прочие косметические правки вне критического пути.

## 2. Текущее состояние

- В `tgbot/bot.py` тяжёлые операции `run_tg_sync_and_ingest`, `run_build_summaries_and_index`, `delete_channel_completely` уже выполняются через `asyncio.to_thread`.
- Вызовы `rag_answer` и `rag_answer_multi` выполняются **синхронно** в async-обработчике — на время RAG цикл обработки апдейтов останавливается для этого worker.
- `utils/cache.py`: ключ кэша строится из нормализованного вопроса и канала(ов), **без** учёта `backend` в `execute_rag_query`.

## 3. Предлагаемое поведение

### 3.1 Telegram-бот

- В месте вызова RAG заменить прямой вызов на:
  - `await asyncio.to_thread(rag_answer_multi, multi_channels, question)` при мультиканальном режиме;
  - `await asyncio.to_thread(rag_answer, normalized_channel, question, mode)` в остальном случае.
- Существующий `try/except` оставить **вокруг** `await`, чтобы исключения из worker-потока обрабатывались как сейчас.
- UX: сообщение «Думаю…», удаление при ошибке, кнопки — без изменения смысла.

### 3.2 Кэш ответов RAG

- Включить в данные для хеша ключа кэша строку **`backend`** (значение, передаваемое в `RAGPipeline`, по умолчанию как сейчас — `"deepseek"`).
- Расширить публичный API класса `Cache` (и при необходимости внутренний `_make_key`): параметр `backend` с значением по умолчанию, чтобы не сломать единичные другие вызовы `get`/`set`/`delete`, если они есть.
- `execute_rag_query`: передавать `backend` в `cache.get` и `cache.set` (и в `delete`, если используется для инвалидации в том же стиле).

### 3.3 Мелкий технический фикс

- В `MemoryCacheBackend` аннотация использует `Dict` — добавить импорт `Dict` из `typing` (или эквивалент), чтобы типы и статический анализ были согласованы. Поведение рантайма не меняется.

## 4. Тестирование

- Добавить или расширить unit-тесты кэша: одинаковые вопрос + канал + `backend` → один и тот же ключ; смена `backend` → другой ключ.
- Запустить `pytest` по затронутым тестам и при необходимости весь набор тестов проекта.

## 5. Риски и обратная совместимость

- После деплоя старые записи in-memory/Redis с прежним ключом не будут совпадать с новым ключом → только **промахи кэша**, не порча данных.
- `to_thread` не меняет детерминизм RAG относительно одного и того же процесса; гонки с глобальным singleton `ChromaStore` те же, что при синхронном вызове из одного процесса.

## 6. Критерии приёмки

- При активном опросе бота другие таймеры/handlers того же процесса не «висят» на время полного RAG цикла (проверка: под нагрузкой или логированием не требуется в спеке; достаточно архитектурного переноса в `to_thread`).
- Смена только `backend` для одного и того же вопроса и канала не возвращает закэшированный ответ от другого backend.
- Все существующие тесты, относящиеся к кэшу и настройкам, проходят; новые тесты на ключ кэша проходят.
